< 設計書を作成する >

その設計書を渡されたエンジニアが画面を作成できるか？ => 目的や機能の実装、動き・処理がハッキリしている！

1.「1シート1コンポーネント」や「1シート1画面」など、設計書をどんな『単位』で切り分けるかをまず決める必要がある。
=> 決め方は、現場によって違う！

2.パラメータ(Param)は、画面遷移時に、受け取る「外部データ」(パラメータ)を書く！
パラメータは、実際に受け渡しをする形を明確に指示する！

3.「DB」から参照する・更新するデータに関する情報を記載する。
    物理名 => DB側での管理名
    論理名 => 日本語で明確化・翻訳
    参照 => データを参照(利用)するかどうか？
    更新 => 追加や削除をするかしないか？

Store経由だとしても、DBデータを参照したり、更新するのならば記載する！

4.項目
「入力」や「動作があるもの」(動きがあるもの)をまとめるをピックアップ！

5.処理 => コンポーネント内の処理ロジックを記載する！
　①タイトル => どういった処理なのか？
　②タイミング => 条件がある場合記載
　③場合分け(パターン・条件分岐) => 「プログラムでどういったことを実現したいのか？」を言語化・明確化する。 

　・特定の外部サービスを利用するような場合は、そのことも指示する。
=> firebaseのログイン認証など

6.DB処理
　・イベント名 => 処理のどのタイミングで呼び出すのかを処理の欄に記載するので注意が必要！
　・種類 => create , update ,delete
　・テーブル名 => データベースへのアクセスパス
　・物理データ名 => DB側での管理名
　・論理名 => 日本語化
　・値
